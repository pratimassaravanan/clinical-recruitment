\title{Adaptive Clinical Recruitment: A Long-Horizon Benchmark for Sequential Trial Planning}

\author{Anonymous Author(s)}

\begin{abstract}
Clinical recruitment creates delayed feedback by construction: screening, follow-up, site allocation, and budget decisions can affect outcomes weeks later, and those outcomes arrive under uncertainty. We present the \textbf{Adaptive Clinical Recruitment} benchmark, a deterministic 180-step environment for sequential trial-planning decisions with three public task difficulties, typed observations, delayed effects, milestone tracking, and site-level constraints. The core benchmark exposes 8 implemented action types and a 37-dimensional numeric feature vector used by the trainable baselines. The repository also contains four repo baselines aimed at different long-horizon behaviors: hierarchical subgoals with hindsight relabeling, milestone-based shaping, multi-scale temporal aggregation, and episodic memory. We rerun the current repository code from the corrected evaluation path and report a fresh 5-seed sweep (30 training episodes per seed, 15 evaluation episodes per seed). The highest mean score in that sweep is 0.2215, but no pairwise comparison reaches $p < 0.05$; under the current training budget, the baseline suite is better read as a sanity check on environment and evaluation behavior than as evidence that one approach clearly wins. The submission artifact includes code, regenerated charts, and a basic passing integration suite for all three tasks.
\end{abstract}

\section{Introduction}

Patient recruitment is one of the main reasons clinical trials miss timelines and overspend budget \citep{fogel2018}. Unlike short-horizon benchmark domains, recruitment decisions are shaped by delayed conversions, dropout risk, site capacity, and non-stationary candidate pools. A benchmark in this space should therefore test more than immediate reward optimization: it should expose planning state, require action-specific candidate selection, and make it easy to reproduce both environment behavior and evaluation outputs.

This paper presents an anonymized NeurIPS E\&D submission package for the \textbf{Adaptive Clinical Recruitment} repository. The aim is not to argue for a new best-performing algorithm, a validated reproduction of any named method, or a deployment-ready simulator. The contribution is a benchmark artifact whose claims stay close to the code, regenerated artifacts, and validation outputs currently present in the repository.

Our main contributions are:

\begin{enumerate}[leftmargin=*]
  \item A deterministic 180-step recruitment environment with three task levels, delayed effects, milestone tracking, site-level metrics, token accounting, and typed observations.
  \item A clearly scoped action interface with 8 implemented action types and action-specific candidate pools for screening, recontact, and allocation.
  \item Four repo baselines that instantiate distinct long-horizon mechanisms using a shared pure-NumPy actor-critic backbone.
  \item A fresh 5-seed sweep, regenerated charts, and 30/30 passing integration checks across the three public tasks.
\end{enumerate}

\section{Benchmark Design}

\subsection{Environment Overview}

The environment simulates a 180-day recruitment window in which an agent manages a patient funnel from screening through enrollment and retention. Figure~\ref{fig:env} summarizes the core benchmark surfaces. At each step, the environment returns a typed observation containing both structured state and a numeric feature representation used by the trainable baselines.

The benchmark provides three public tasks with increasing difficulty (Table~\ref{tab:tasks}). The harder tasks tighten budget, increase target enrollment, and add more site coordination pressure.

\subsection{Observations, Actions, and Rewards}

The training stack extracts a 37-dimensional numeric feature vector from each observation. That vector is built from timestamp, budget, deadline, enrollment progress, funnel counts, milestone status, constraint flags, uncertainty components, memory summaries, and counterfactual rollout hints. The full typed observation also carries structured fields such as \texttt{available\_patients}, \texttt{recontact\_candidates}, \texttt{allocation\_candidates}, \texttt{site\_performance}, \texttt{current\_plan}, and \texttt{retrieved\_memory\_context}.

The current action interface contains 8 implemented action types (Table~\ref{tab:actions}). This is the interface used by \texttt{models.py}, \texttt{env.py}, and \texttt{training/neural\_policy.py}; earlier drafts that listed additional actions were incorrect and are not used in this submission.

\begin{table}[t]
  \centering
  \caption{Implemented action interface. Candidate selection is action-specific.}
  \label{tab:actions}
  \small
  \begin{tabularx}{\linewidth}{lX}
    \toprule
    Action & Primary role \\
    \midrule
    \texttt{screen\_patient} & Screen a candidate from \texttt{available\_patients}. \\
    \texttt{recontact} & Re-engage a candidate from \texttt{recontact\_candidates}. \\
    \texttt{allocate\_to\_site} & Allocate a consented candidate from \texttt{allocation\_candidates}. \\
    \texttt{adjust\_strategy} & Change outreach, criteria, or site focus. \\
    \texttt{plan\_next\_phase} & Update high-level plan state for the next phase. \\
    \texttt{summarize\_and\_index} & Write a compact entry into indexed memory. \\
    \texttt{retrieve\_relevant\_history} & Query indexed memory for current bottlenecks. \\
    \texttt{stop\_recruitment} & Terminate the episode early. \\
    \bottomrule
  \end{tabularx}
\end{table}

Rewards combine screening success, enrollment gain, dropout penalty, budget efficiency, timeline pressure, milestone events, hypothesis accuracy, and token efficiency. Delayed effects are queued explicitly, so actions can trigger downstream consequences several steps later.

\subsection{Core Scope Versus Experimental Extras}

The repository contains additional research helpers beyond the core benchmark path, including offline analysis modules, counterfactual simulation utilities, curriculum managers, and reporting stubs. In this paper, we treat only the following as the code paths directly supporting the paper's claims: \texttt{env.py}, \texttt{models.py}, the four trainable baselines, \texttt{experiments/full\_sweep.py}, \texttt{experiments/reproducibility.py}, and the test scripts used for validation. We do \emph{not} treat modules such as \texttt{training/async\_rl.py}, \texttt{training/progressive\_rl.py}, or \texttt{experiments/appendix\_report.py} as benchmark-backed contributions.

\section{Repository Baselines and Evaluation Protocol}

\subsection{Repo Baselines}

The repository includes four trainable baselines built on a shared pure-NumPy actor-critic implementation. We describe them as repo baselines rather than strict reproductions or faithful implementations of external named methods.

\begin{table}[t]
  \centering
  \caption{Repo baselines and the long-horizon mechanism each emphasizes.}
  \label{tab:baselines}
  \begin{tabularx}{\linewidth}{lX}
    \toprule
    Baseline & Main mechanism \\
    \midrule
    HCAPO & Repo baseline with planner/executor separation and hindsight relabeling, borrowing the relabeling idea from HER \citep{andrychowicz2017hindsight}. \\
    MiRA & Repo baseline with a learned milestone potential term, borrowing the reward-shaping idea from policy-invariant shaping \citep{ng1999policy}. \\
    KLong & Repo baseline with overlapping temporal windows and TD($\lambda$)-style credit flow \citep{sutton2018}. \\
    MemexRL & Repo baseline with gated episodic writes and similarity-based retrieval, borrowing from neural episodic control \citep{pritzel2017}. \\
    \bottomrule
  \end{tabularx}
\end{table}

All four baselines use the same 37-dimensional feature extractor from \texttt{training/neural\_policy.py}. Before rerunning the reported sweep, we corrected the evaluation path so that \texttt{recontact} and \texttt{allocate\_to\_site} choose patients from their action-specific candidate pools instead of from the unscreened screening pool. All results reported below were regenerated after that code-path correction.

\subsection{Training and Compute}

The fresh sweep uses 5 seeds: \{1, 7, 21, 42, 123\}. For each seed and baseline, \texttt{experiments/full\_sweep.py} runs 30 training episodes while cycling across the three tasks, then evaluates on 5 episodes per task (15 evaluation episodes total). Pairwise comparisons use paired $t$-tests, Wilcoxon signed-rank tests, Cohen's $d$, and bootstrap 95\% confidence intervals computed in \texttt{experiments/reproducibility.py}.

All reported experiments were executed locally on a 12th Gen Intel Core i5-1245U CPU with 16~GB RAM, Python~3.14, and no GPU. The full 4-baseline $\times$ 5-seed sweep completed in approximately 295.5 seconds (4.9 minutes) end-to-end.

No pairwise comparison reaches $p < 0.05$. The smallest $p$-value is 0.1823 for HCAPO versus MiRA. HCAPO versus KLong, a comparison overstated in older drafts, now yields $p = 0.3849$. With only five seed-level observations per baseline, these tests are low-power and descriptive. The current data do not support claims of statistical separation or a clear winner; they show that the four repo baselines land in a similar score range under this budget.

The integration suite passes 10/10 checks on each of \texttt{easy\_bench}, \texttt{medium\_bench}, and \texttt{hard\_bench}, for a total of 30/30 checks. These are 20-step smoke checks for observation structure, site metrics, milestone exposure, plan and memory fields, token-budget state, multi-step execution, action diversity, and non-zero reward accumulation. They are useful for catching interface regressions, but they are not evidence of full behavioral correctness, clinical realism, or deployment readiness.

\section{Interpretation and Limitations}

The current repository is strongest as a compact evaluation artifact: the environment interface is typed and deterministic, the action construction path is now consistent with the observation schema, and the reported figures can be rerun from a short CPU job. The benchmark is less mature as an empirical leaderboard. Baseline means are close, variance remains non-trivial, and the present training budget is small.

Several limitations matter for reviewers. First, the environment is synthetic and should not be treated as a deployment proxy for actual clinical operations. Second, some repository modules remain scaffolds or offline research helpers rather than benchmark-backed components. Third, the training stack uses simple pure-NumPy neural networks rather than stronger modern RL backbones. Finally, five seeds and 30 training episodes per seed are enough to expose code-path issues, but not enough to support strong ranking claims; stronger evaluation protocols (more seeds, larger training budgets, and ablations over individual state channels) are still needed.

The broader impact is mixed. A workflow benchmark can help stress-test long-horizon agents for planning, memory, and delayed effects, but any real-world use in healthcare operations would raise questions about fairness, patient autonomy, and inappropriate automation. For that reason, we position this artifact as a research benchmark only.

\section{Conclusion}

The \textbf{Adaptive Clinical Recruitment} repository provides a long-horizon benchmark artifact with a deterministic 180-step environment, a corrected 8-action interface, four repo baselines, fresh 5-seed sweep outputs, and passing integration checks across three task difficulties. The current evidence does \emph{not} show statistically significant separation among the baselines, and it should not be read as establishing a winner. Future work on this benchmark should emphasize stronger training protocols, sharper ablations, and a continued separation between core benchmark claims and experimental scaffolds.
